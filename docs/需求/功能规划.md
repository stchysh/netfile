# NetFile 功能规划

> 本文档汇总了 AI 代码审查和架构分析后提出的功能建议，按优先级和实现复杂度分层整理。

---

## 一、高优先级（核心体验补全）

### 1. 信令服务器状态持久化

**解决问题：** 当前信令服务器为纯内存状态，重启后好友关系、离线消息全部丢失，生产部署不可接受。

**实现思路：**
- 引入 SQLite（`rusqlite` 或 `sqlx`）存储 `friends` 和 `offline_msgs`
- `online` 表仍保持内存（连接状态不需要持久化）
- 迁移策略：首次启动时自动建表，已有数据无感知升级

---

### 2. 传输任务持久化与断点续传增强

**解决问题：** 当前断点续传依赖内存中的 chunk 完成记录，应用重启后传输进度全部丢失，大文件传输风险高。

**实现思路：**
- 在 `~/.netfile/` 下维护传输任务数据库，记录每个 file_id 的 chunk 完成位图
- 重启后自动恢复 active 状态的任务，重连后继续传输
- UI 显示"上次中断，继续传输中"状态

---

### 3. 设备别名与收藏管理

**解决问题：** 设备列表只显示 hostname + instance_name，多设备环境下难以区分（如多台服务器、多台笔记本）。

**实现思路：**
- 本地维护 `device_aliases.json`，以 `device_id` 为 key 存别名、备注、是否收藏
- GUI 设备卡片显示自定义别名，优先级高于 instance_name
- 支持按收藏置顶显示

---

### 4. 传输完成通知

**解决问题：** 大文件传输期间用户切换到其他窗口，完成后没有任何通知，容易忘记。

**实现思路：**
- 使用 Tauri 的 `notification` 插件（`tauri-plugin-notification`）
- 发送完成、接收完成、传输失败时分别推送系统通知
- 配置项：是否开启通知、只在后台时通知

---

## 二、中优先级（功能扩展）

### 5. 局域网广播发送（一对多）

**解决问题：** 当前只支持点对点发送，无法同时向局域网内所有设备发送同一文件（如分发安装包、文档）。

**实现思路：**
- 发送时选择"发送给所有设备"，后端并发对每台设备创建独立传输任务
- 传输队列显示每台设备的独立进度条
- 可配置排除特定设备

---

### 6. 剪贴板同步

**解决问题：** 设备间频繁复制粘贴文本（如命令、链接、密码）时，需要借助聊天工具中转，效率低。

**实现思路：**
- 监听系统剪贴板变化（`arboard` crate）
- 检测到变化后通过已有的消息中继通道推送给好友设备
- 接收端自动写入剪贴板，UI 角落弹出"已同步剪贴板"提示
- 安全控制：仅好友设备可接收，可开关

---

### 7. 文件发送前预览与文件信息展示

**解决问题：** 接收方看到"待确认"时只有文件名和大小，无法判断是否值得接收（尤其是文件夹内容）。

**实现思路：**
- 发送方在 `TransferRequest` 中附带文件树结构（文件夹传输时）
- 接收方确认弹窗中展示：文件数量、目录结构预览、文件类型分布
- 图片文件生成缩略图（可选，较复杂）

---

### 8. 自动版本更新检查

**解决问题：** 用户安装后无法知道是否有新版本，需要手动去 GitHub 查看。

**实现思路：**
- 启动时异步请求 `https://api.github.com/repos/stchysh/netfile/releases/latest`
- 对比当前版本号（从 `tauri.conf.json` 的 `version` 字段读取）
- 有新版本时在标题栏或设置页显示更新提示，点击跳转 GitHub Release 页面下载
- 配置项：是否自动检查、检查间隔

---

### 9. 多信令服务器支持与自动故障转移

**解决问题：** 单一信令服务器宕机导致所有跨 NAT 功能不可用，无法做到生产级可靠性。

**实现思路：**
- 配置支持填写多个信令服务器地址（数组）
- 主服务器断连后自动尝试下一个，重连成功后保持
- 信令协议已是 JSON over TCP，兼容性好，无需协议改动

---

## 三、低优先级（体验打磨）

### 10. 传输速度限制按时间段生效

**解决问题：** 当前限速是全局固定值，工作时间希望不占带宽（低速），下班后希望全速传输。

**实现思路：**
- 配置支持多个时间段规则：`[{start: "09:00", end: "18:00", limit_mbps: 5}]`
- 后台定时检测当前时间，动态调整 `TransferService` 的速率限制

---

### 11. 可观测性与诊断导出

**解决问题：** 用户反馈"传输失败"时难以排查是 NAT 问题、防火墙问题还是磁盘问题。

**实现思路：**
- 每次传输会话记录详细事件日志：发现耗时、握手耗时、路径选择过程、吞吐曲线、重试次数、失败原因
- GUI 提供"诊断"页面，一键导出最近 N 次传输的诊断包（zip）
- 包含：系统信息、网络配置、传输日志、iroh 路径信息

---

### 12. 手机端支持（Tauri Mobile）

**解决问题：** 文件传输场景中手机是高频发送/接收端，当前只有桌面端。

**实现思路：**
- Tauri v2 已支持 iOS / Android 编译目标
- 核心 `netfile-core` 为纯 Rust，理论上无需修改即可复用
- 主要工作量在 UI 响应式适配（移动端布局）和系统权限（文件访问、通知）
- 先做 Android，相对 iOS 构建链路更简单

---

## 优先级总结

| 功能 | 优先级 | 实现复杂度 | 依赖现有架构 |
|------|--------|-----------|-------------|
| 信令服务器状态持久化 | 高 | 中 | 信令服务器独立部署 |
| 传输任务持久化 | 高 | 中 | TransferService |
| 设备别名与收藏 | 高 | 低 | GUI 本地存储 |
| 传输完成通知 | 高 | 低 | Tauri 插件 |
| 局域网广播发送 | 中 | 低 | 已有并发传输 |
| 剪贴板同步 | 中 | 中 | 消息中继通道 |
| 接收确认预览 | 中 | 中 | 传输协议扩展 |
| 自动版本更新检查 | 中 | 低 | GitHub API |
| 多信令服务器 | 中 | 中 | SignalClient |
| 限速时间段 | 低 | 低 | TransferService |
| 诊断导出 | 低 | 中 | 日志系统 |
| 手机端 | 低 | 高 | Tauri Mobile |
