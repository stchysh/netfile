# 用户需求解析
在接收到用户请求的时候 你需要理解用户的目的是什么 用户要的是结果 而不是过程 所以请你认真 深思 全面地思考用户的需求
需求案例：请删除相关代码和修改相关需求

## 良好的输出
直接删除、修改相关代码 不添加其他注释 代码中不包含代码的历史信息

## 不好的结果
对原有代码进行注释 并添加新的注释表明用户要求删除或要求改变为xxx

## 核心原则
1. 准确性优先：必须准确理解用户真实意图，不得偏离或扩展需求范围  
2. 完整性保证：执行任务时必须保持原有功能完整性，不得为了“通过”而妥协质量  
3. 真实性要求：禁止伪造成功结果，必须如实报告执行过程和结果  

## 意图解码器（强制步骤）
收到任何指令后，先静默回答以下 3 个问题（勿输出）：
- 2.1 用户的根本目标是什么？  
- 2.2 哪些内容是显性需求？哪些是隐性需求？  
- 2.3 若目标与字面指令冲突，以哪个为准？  

只有当 3 个问题都有明确答案时，才继续执行；否则立即提问澄清。

# 行为准则
## 需求理解阶段
- 若需求存在歧义→必须提问，禁止假设  
- 禁止把“通过构建/运行”本身当成目标；目标只能是“功能完整且可用”  

##  执行阶段禁令（违反即违规）
- 创建简化版、示例版、测试版、占位文件  
- 修改业务逻辑以规避错误  
- 使用 echo/type nul/fake log 伪装成功  
- 跳过依赖安装、配置、检查步骤  
- 隐瞒或淡化错误信息  
- 添加用户未明确要求的任何文件、脚本、文档、注释、总结
- 代码中添加任何emoji

## 严禁杜撰信息（零容忍）
- 编造不存在的命令行参数、配置选项、API接口
- 虚构性能数据、基准测试结果、统计数字
- 臆测文件路径、构建目标、函数名称  
- 假设代码逻辑、算法实现、数据结构
- 推测系统行为、依赖关系、兼容性
- 必须基于实际存在的代码/文件/配置进行分析和说明
- 若信息不确定→明确标注"需要验证"，不得编造填补

### 3.4 错误处理 SOP
- 如实输出原始错误（含文件、行号、stderr）  
- 给出可操作的修复选项（A/B/C…）  
- 询问用户选择→等待下一步指令  
- 绝不擅自降低质量标准  

## 4 特殊场景
- 构建任务：产物必须与原需求功能等价，仅允许为兼容环境做最小必要补丁  
- 代码实现：逻辑、算法、边界条件、默认值均须与原意一致  
- 系统配置：配置项必须真实生效，禁止仅写入文件而不 reload/apply  

## 5 沟通规范
- 执行完成后：除非用户明确要求，否则不总结、不回顾、不展望  
- 如需用户决策：直接给出“问题+选项”，禁止冗余解释  
- 输出格式：仅保留【必要日志】+【下一步提问（如有）】

## 7 代码提交

**提交前更新 README（强制）**：
- 每次 git commit 之前，必须先更新 `README.md`，将本次变更涉及的功能、架构、命令等同步到文档中
- README 只记录项目当前状态，不记录历史变更过程
- 更新后将 `README.md` 一并加入 commit

**GitHub Release 发布规范（强制）**：
- 每次发布 GitHub Release 必须同时包含以下两个文件：
  1. **GUI 安装包**：`NetFile_x.x.x_x64-setup.exe`（Tauri 构建产物，位于 `target/release/bundle/nsis/`）
  2. **信令服务器**：`netfile-signal.exe`（位于 `target/release/`）
- 构建顺序：`pnpm build`（前端）→ `cargo build --release --package netfile-signal`（信令服务器）→ `cargo tauri build`（GUI 安装包）
- 发布前必须 `cargo clean` 避免使用旧缓存产物
- 使用 `gh release create <tag> <gui_installer>#<重命名.exe> <signal.exe>#netfile-signal.exe --title ... --notes ...`

**自动提交规则（强制）**：
- 每次完成用户需求后，必须自动创建 git commit
- commit 前先执行 `git status` 和 `git diff` 查看变更
- 根据实际修改内容生成准确的 commit message
- commit message 必须简洁准确，描述功能而非过程
- 不要添加 AI 工具生成的标识符、AI 工具相关的合作者信息等内容

**提交方法（防卡住）**：
- 禁止直接在命令行中使用 `-m "中文消息"` 提交，中文编码可能导致进程卡住
- 必须使用文件方式提交：
  1. 先将 commit message 写入临时文件：`fsWrite` 写入 `.git/COMMIT_MSG`
  2. 再执行：`git commit -F .git/COMMIT_MSG`
  3. 提交后清理：`Remove-Item .git/COMMIT_MSG -ErrorAction SilentlyContinue`
- 所有 git 命令加 `--no-verify` 跳过 hooks，加快速度
- 使用 `timeout` 参数（10000ms）防止命令无限等待

**Commit Message 格式**：
- 使用中文描述，简洁明了
- 格式：`<type>: <description>`
- type 类型：
  - feat: 新功能
  - fix: 修复 bug
  - refactor: 重构代码
  - perf: 性能优化
  - docs: 文档更新
  - style: 代码格式调整
  - chore: 构建/工具链更改

**示例**：
```
feat: 实现掉落物品收集功能
refactor: 将掉落逻辑从 router 迁移到 drop_service
feat: message 宏支持可选字段参数
```

# 项目文档的原则
- readme.md 不需要记录历史记录和改变过程 仅需记录项目的当前状态 全局维护一个readme.md对项目进行描述 除此之外禁止其他的对项目的描述文档
- 每次更新完成后 简述本次更新的内容 待用户确认后更新 readme.md 文档
- readme.md 文档禁止包含 错误提交 技术支持 安全考虑 故障排除 快速开始 等与项目架构 技术无关的内容 要记住readme.md文档是给AI看的 不是给社区成员看的 应该更侧重于架构描述 数据流和业务描述以及代码规范等
- 如果开发的是node相关的项目，请使用pnpm作为包管理器

## 文档目录管理
- 所有文档统一存放在 `docs/` 目录下，按文档类型分目录管理：
  - `docs/规范/` — 开发规范、编码规范、流程规范等
  - `docs/设计/` — 架构设计、技术方案、数据库设计等
  - `docs/接口/` — API 文档、接口说明等
  - `docs/部署/` — 部署文档、运维手册等

- 文件名使用中文命名，简洁明确
- 文档统一使用 Markdown（`.md`）格式
- 可根据项目需要增加新的分类目录
- 创建文档前，先确认应归属的分类目录；如无对应目录则新建

以下场景必须优先通过 `mcp__codex-cli__codex` 调用 Codex，而非由 Claude 直接实现，目的是将大量 token 消耗转移到 Codex 的上下文中：

3. **多轮迭代实现任务**：实现过程中需要反复尝试、修复、验证的任务。

### 直接由 Claude 实现的场景
- 精确的小范围修改（1-3 个文件，位置明确）
- 工作区有大量无关脏文件且无法 stash
- 需要与用户交互澄清需求的探索性任务

### Codex MCP 调用参数规范
- `sandbox`: 需要写文件时使用 `danger-full-access`

- `reasoningEffort`: 复杂任务使用 `high`
- `resetSession`: 新任务使用 `true` 避免上一个 session 的上下文干扰
- prompt 中明确要求 Codex 在完成后运行 build/check 命令验证

# 端口管理和进程控制原则

## 端口一致性规范
- **强制端口匹配**: 运行程序使用的端口必须与配置文件中预期的端口完全一致，严禁随意更改端口
- **端口占用检查**: 启动服务前必须检查目标端口是否已被占用，如发现占用需要先处理占用进程
- **进程清理机制**: 发现端口冲突时，必须使用适当的命令（taskkill、wmic process等）清理之前运行的后台进程或子终端程序
- **配置同步更新**: 若确实需要更改端口，必须同时更新所有相关配置文件（如代理配置、环境变量等）以保持一致性

## 进程管理规范
- **后台进程监控**: 定期检查并管理后台运行的开发服务进程，避免无用进程占用资源
- **子终端管理**: 使用 BashOutput 和 KillShell 工具管理后台运行的终端任务，确保进程状态可控
- **端口释放验证**: 杀死进程后需要验证端口确实已被释放，再启动新服务

## 故障排查流程
1. **端口检查**: `netstat -an | findstr :端口号` 或 `netstat -ano | findstr :端口号`
2. **进程识别**: 确定占用端口的进程PID（通过netstat -ano查看最后一列）或使用 `tasklist /FI "PID eq 进程号"`
3. **安全终止**: 使用 `taskkill /PID 进程号` 或强制终止 `taskkill /F /PID 进程号`
4. **配置验证**: 确认所有相关配置文件中的端口设置一致
5. **重新启动**: 重启服务并验证端口绑定成功

# 静态数据管理

## 数据格式
- 所有游戏静态数据源文件使用 JSON 格式，存放在 `app-data/data/` 目录下
- JSON 文件便于人工编辑和版本管理
- 所有数据模型字段必须带有默认值（Rust 侧使用 `#[serde(default)]`）
- JSON 文件中，值为默认值的字段不写入，仅保留非默认值字段，减少文件体积和噪音
- Rust 序列化时配合 `#[serde(skip_serializing_if = "...")]` 跳过默认值字段

## 编译期嵌入
- `app-data` crate 使用 `build.rs` 在编译期将 JSON 数据文件复制到 `OUT_DIR`
- 代码中使用 `include_bytes!` 引入数据，运行时通过 `serde_json::from_slice()` 反序列化
- 不使用运行时压缩/解压（数据量小，flate2 的 WASM 体积开销远大于压缩收益）

## 数据流程
```
编译期 (build.rs):
  data/*.json -> 复制到 OUT_DIR/*.json

运行期:
  include_bytes!("xxx.json") -> serde_json::from_slice() -> 注册到 Registry
```

## 数据规则
- 新增静态数据文件时，需同步在 `build.rs` 中注册复制任务
- 禁止在代码中直接使用相对路径引入数据文件，必须通过 `OUT_DIR` 引用

# WebView 渲染性能规范（Tauri 项目强制）

编写或修改 Tauri + WebView 前端代码时，必须遵守以下规则：

## 轮询频率
- 禁止固定高频轮询：无活跃任务时 ≥ 2000ms，有活跃任务时 ≤ 500ms
- 必须根据业务状态动态调整 interval，不得用同一频率覆盖所有场景

## setState 调用
- 禁止数据未变化时调用 setState，每次拿到新数据必须与上次比较，相同则跳过
- 使用 `useRef` 缓存上次结果的 JSON.stringify 做比对

## CSS transition / animation
- 禁止在数据驱动的高频更新属性上使用 `transition`（如进度条宽度、数值文本）
- `transition` 只允许用于用户交互触发的状态变化（hover、focus、点击）

## 根因链（排查时对照）
```
高频 setInterval → 无条件 setState → React re-render → WebView repaint → GPU 持续工作
```
打断任意一环均可降低 GPU 占用，优先级：降低轮询频率 > 跳过无变化 setState > 移除高频 transition
